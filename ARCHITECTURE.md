# Architecture

How the Quran Khatma Tracker is organized, and the conventions that keep it easy
for a future collaborator to pick up. This reflects the design mandated by
[REQUIREMENTS.md](REQUIREMENTS.md) §3.

## Overview

- A **static, framework-free** TypeScript app built by Vite into two HTML
  entries (member + admin) that share all code under `src/`.
- The browser talks **directly to Firestore** (client SDK). There is no server
  and no Cloud Functions — the admin's browser runs all privileged logic.
- Code is split into **layers with a strict, one-directional dependency rule**,
  enforced by ESLint so the boundaries can't quietly erode.

## Layers

```
        content/  ── strings + Quran text (no logic)
           │
theme/     │        ui/  ── rendering + event wiring (no business logic)
  │        │         │  │
  │        ▼         ▼  ▼
  └──▶  domain/ ◀── data/  ── Firestore reads/writes (the ONLY Firebase importer)
        (pure)
```

| Layer           | Path           | Responsibility                                                        | May import                           |
| --------------- | -------------- | --------------------------------------------------------------------- | ------------------------------------ |
| **Data-access** | `src/data/`    | Every Firestore read/write, as typed functions + realtime subscribers | `firebase/*`, `domain` types         |
| **Domain**      | `src/domain/`  | Pure logic: assignment, progress math, validation, types              | nothing (must stay pure)             |
| **Content**     | `src/content/` | All Arabic UI strings + the bundled Quran dataset                     | (data only)                          |
| **Theme**       | `src/theme/`   | Design tokens (CSS) + the reading-scale helper                        | —                                    |
| **UI**          | `src/ui/`      | Build DOM, wire events, call `data` + `domain`                        | `data`, `domain`, `content`, `theme` |

### Enforced boundaries (guardrails)

These are real ESLint rules in [`eslint.config.js`](eslint.config.js), not just
conventions — `npm run lint` fails if they're broken:

1. **Firebase is importable only inside `src/data/`.** Every other layer must go
   through the typed data-access functions. This keeps Firestore details in one
   place and makes the rest of the app testable without a database.
2. **The domain layer must stay pure** — it cannot import `firebase`, `data`, or
   `ui`. This guarantees the assignment algorithm and calculations are unit-
   testable in isolation and free of side effects.

## Firestore data model

Collections and their document shapes (typed in
[`src/domain/types.ts`](src/domain/types.ts)):

| Path                                        | Type            | Notes                                            |
| ------------------------------------------- | --------------- | ------------------------------------------------ |
| `roster/{personId}`                         | `Person`        | Global roster; `name` unique; `completedPages[]` |
| `khatmas/{khatmaId}`                        | `Khatma`        | `memberIds[]`, `totalPages` (604), `anonymous`   |
| `khatmas/{khatmaId}/assignments/{memberId}` | `Assignment`    | `pagesByDay[day][]`, `doneByDay{day: ts}`        |
| `content/global`                            | `GlobalContent` | Single admin-editable `du3aText`                 |

Key behaviors this shape is designed to support (implemented in later stages):

- **Today's pages** — active khatmas containing me → my `assignments` doc → the
  day index derived from `startDate`.
- **One-tap "done"** — a batch write that sets `doneByDay[day]` **and** appends
  that day's pages to `Person.completedPages` (deduped). Only pages marked done
  ever enter the rotation history (REQUIREMENTS §5).
- **Rotation** — the assignment algorithm reads each member's `completedPages`
  to avoid reassigning pages they've already finished, so a person progressively
  covers the whole Quran across khatmas.
- **Insights** — lifetime % = `completedPages.length / 604`; khatma % =
  done pages / `totalPages` (`src/domain/progress.ts`).
- **Admin correction** — clearing `doneByDay[day]` and removing those pages from
  `completedPages` fixes an accidental wrong-name tap.

Composite indexes, if any become necessary, are tracked in
[`firestore.indexes.json`](firestore.indexes.json).

## Content & theme conventions

- **No hardcoded copy.** All user-facing Arabic text lives in
  [`src/content/strings.ar.ts`](src/content/strings.ar.ts); HTML files carry no
  copy and the entries even set `document.title` from strings. Add/adjust wording
  there, never inline.
- **One place to restyle everything.** Design tokens (colors, radii, fonts) are
  Tailwind v4 `@theme` variables in
  [`src/theme/theme.css`](src/theme/theme.css). Each token is both a utility
  (`--color-primary` → `bg-primary`) and a CSS variable — edit tokens to
  re-skin the whole app without touching pages. This is developer-only; the admin
  never edits styling.
- **RTL first.** Both entries set `dir="rtl" lang="ar"`; prefer logical utilities.
- **Reading scale.** The 5-level font-size setting
  ([`src/theme/reading.ts`](src/theme/reading.ts)) sets `data-reading-scale` on
  `<html>`, which the CSS maps to `--reading-scale`; `.quran-text` sizes from it.

## Quran data

The full Madinah mushaf (King Fahd / KFGQPC, Hafs Uthmani) is bundled as
page-accurate JSON under `public/quran/`, generated once by
[`scripts/build-quran-data.ts`](scripts/build-quran-data.ts) from the quran.com
API and committed — the app has **no runtime Quran-API dependency**.

- **Pages are the atomic unit** (matching how assignment works). `pages/NNN.json`
  is a `QuranPage` (ayat with surah/ayah/text/sajda); `surahs.json` gives each
  chapter's first/last page so the admin can assign a **page range, a whole
  chapter, or both**; `index.json` maps surah/juz → page ranges.
- **Access** is through [`loader.ts`](src/content/quran/loader.ts) (async,
  cached `fetch` keyed off `import.meta.env.BASE_URL`), so it works at any base
  path. Text renders reflowably (scales with the reading slider) using the
  bundled Amiri Quran font, with ayah medallions from
  [`symbols.ts`](src/content/quran/symbols.ts). Swap in the full 604-page reading
  UI in a later stage without touching this data layer.

## Security

**There is no authentication in this app** — this is a deliberate, accepted
trade-off for a trusted community of senior users (REQUIREMENTS §3, §4, §8).
Understand exactly what that means:

- The Firebase web config ships in the **public client bundle**. It is not a
  secret and cannot be hidden.
- Firestore security rules ([`firestore.rules`](firestore.rules)) can therefore
  only **validate document shape and reject unknown paths** — they are **not an
  identity boundary**. Anyone who finds the config could write to the database.
- The admin app is protected only by an **unguessable URL**
  (`admin-<slug>.html`), never linked from the member app. This is obscurity,
  not access control.
- Member identity is a **trust-based** localStorage cache
  ([`src/ui/shared/identity.ts`](src/ui/shared/identity.ts)); anyone could tap a
  wrong name. The admin can correct mistaken marks.

Do not add features that assume these rules protect data by identity. If real
protection is ever needed, it requires introducing authentication (out of scope
for v1) and rewriting the rules around it.

## Build & deploy model

- **Vite MPA** — two `rollupOptions.input` entries in
  [`vite.config.ts`](vite.config.ts) produce one bundle per HTML page from
  shared `src/`. `base` is env-driven (`BASE_PATH`) for GitHub Pages project
  paths. Firebase lands only in the member bundle (the admin placeholder doesn't
  import it), so bundles stay layer-appropriate.
- **Static hosting** — the whole thing is static files on GitHub Pages. Firestore
  rules deploy separately via the Firebase CLI.
- **Local dev** uses the **Firestore emulator** (dev builds auto-connect via
  `import.meta.env.DEV`), so building never touches real data.

## Testing

- The **domain layer is the test priority** because it's pure. Vitest runs in a
  Node environment against `tests/**` (see
  [`tests/domain/validation.test.ts`](tests/domain/validation.test.ts)).
- Data and UI layers are kept thin so most logic worth testing lives in `domain`.
