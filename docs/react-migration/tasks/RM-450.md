## Scope

RM-400 through RM-440 each shipped a per-feature member suite that exercises its
own route from a preset state (`member-identity`, `member-khatma-routes`,
`member-personal-settings`, `member-completion`, `member-reader`). RM-450 adds
the dedicated **integration** layer: continuous, cross-feature journeys through
the composed member tree (`MemberIdentityBoundary` → `MemberExperience`, i.e.
assignments subscriptions + completion interrupt + shell + routes), proving the
Phase 4 exit criterion that "the React member preview has functional parity and
realtime updates do not reset reader state" holds when everything is wired
together — not re-testing each feature's internal states.

New suite: `tests/app/member-integration.test.tsx`.

## Acceptance criteria

- **Critical route journey (gate → list → landing → reader → finish):** with no
  remembered identity, the real gate selects a member, the shell appears, and the
  user navigates list → khatma landing → assigned reader and finishes the round —
  one continuous flow across four feature boundaries.
- **Realtime across route changes:** a live assignment tick received on one route
  is reflected after navigating to another, through the single shared store
  listener, with no duplicate subscription created by navigation.
- **Identity switch teardown/re-subscribe:** switching person from the shell
  returns to the gate and, on picking a different member, subscribes that
  member's khatma assignments while the previous member's listener is released.
- **Completion interrupt over the reader:** an unacknowledged completed khatma
  overlays the assigned reader route; acknowledgement restores the underlying
  reader with its page content and navigation.

## Decisions and risks

- The suite mounts `<MemberIdentityBoundary><MemberExperience/></MemberIdentityBoundary>`
  through `renderWithAppProviders`, which is the full member tree minus the outer
  `AppProviders` supplied by the harness — the established member-suite pattern.
- The Quran loader (`@/content/quran/loader`) is mocked exactly as in
  `member-reader.test.tsx` so reader routes stay deterministic and offline.
- Scenarios are chosen to cross feature boundaries the per-task suites do not,
  avoiding duplication of already-covered single-route states.
