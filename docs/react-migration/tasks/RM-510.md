RM-500 admin shell, replacing the navigable placeholder with the real page.

Parity oracle:
[`REACT_MIGRATION_UI_INVENTORY.md`](../../../REACT_MIGRATION_UI_INVENTORY.md)

[`src/ui/admin/pages/roster.ts`](../../../src/ui/admin/pages/roster.ts).

In scope:

- **Search-as-you-type** over the roster by name substring, with caret/focus
  preserved across the keystroke re-render (**P4** — a controlled MUI field that
  stays mounted does this without the legacy's manual re-focus hack).
- **Per-person row:** name (strike-through + muted when disabled),
  `disabledBadge`, a pages/round **stepper** (± ; min 1) → `updatePerson`,
  `disable`/`enable` toggle → `updatePerson`, `remove` → confirm
  `confirmRemove` → `removePerson`.
- **Add form:** name, optional note, pages/round number, `addPerson` button;
  client-side validation `nameRequired` (blank) and `nameTaken`
  (`isNameUnique`); resets name/note/error on success (keeps pages/round, as
  legacy does).
- **States:** empty roster → `emptyRoster`; query with no match → `noMatches`.

Out of scope: Khatmas list/create (RM-520), Khatma detail (RM-530), Settings
(RM-540), the form-draft stability proof (RM-550), and the full admin
integration suite (RM-560). RM-510 ships focused Roster tests only.

## Approach

- New `src/app/admin/pages/RosterPage.tsx` (`AdminRosterPage`); wire it into
  `AdminApp`'s `AdminRouteContent` (`roster` → `AdminRosterPage`, dropping the
  placeholder for that route).
- Reuse the shared primitives: `SurfaceCard`, `AppTextField` (search + add
  fields), `NumberStepper` (row pages/round), `AppButton`, `StatusChip`
  (disabled badge), `useConfirmation()` (remove), `useWriteOperation()`
  (`addPerson` / `updatePerson` / `removePerson`).
- **Drafts are component-local `useState`.** Because the page stays mounted while
  live snapshots re-render it, search text and the add-form fields survive those
  snapshot re-renders (the P2/P4 parity concern). The full draft-stability proof
  is RM-550.
- **Feedback granularity (§5 quirk 5):** stepper `pagesPerDay`, enable/disable,
  and the `addPerson` write are fire-and-forget (no success/error surface); only
  the client-side add validation (`nameRequired`/`nameTaken`) is shown. Match the
  legacy exactly.

## Acceptance criteria

- Search filters by name substring as-you-type and preserves caret/focus (P4).
- Row controls (stepper min 1, enable/disable, remove-with-confirm) drive the
  correct `updatePerson`/`removePerson` writes; disabled rows are visually muted
  and badged.
- Add form validates blank/duplicate names, adds a unique person, and resets
  name/note on success.
- Empty roster and no-match states match the legacy copy.
- Typecheck, lint, and the full test suite pass (Roster tests added).

## Intentional deltas (recorded per inventory §1.7 / §5)

1. **P4 without the re-focus hack** — the legacy rebuilds the whole DOM per
   keystroke and then manually re-focuses the search input
   (`rerenderPreservingSearchFocus`). The React port keeps the controlled search
   field mounted, so the browser preserves caret/focus natively; the visible
   behavior is identical.
2. **Draft scope** — search text and the add-form fields are component-local
   state that survives live-snapshot re-renders (the parity-critical case) but is
   route-scoped rather than held in a single app-lifetime `AdminDraft`. The
   snapshot-survival behavior the inventory calls out is preserved; RM-550
   formalizes the draft-stability model across all admin forms.
3. **Add-validation a11y** — the blank/duplicate-name message carries
   `role="alert"` and an error tone (the legacy shows it as plain danger text),
   so it is announced when it appears. No copy or trigger change.
4. **Remove confirmation** — the destructive remove uses the shared confirmation
   dialog on a `danger` tone (the RTL-safe replacement for `window.confirm`).

No functional, data, domain, routing, or validation behavior was changed.

## Implementation

- New `src/app/admin/pages/RosterPage.tsx` (`AdminRosterPage`):
  - `AppTextField type="search"` bound to component-local `search` state; the list
    filters by name substring (`name.includes(query.trim())`), matching the legacy
    filter exactly. The controlled field stays mounted, so caret/focus survive the
    keystroke re-render (**P4**) natively.
  - `PersonRow` — name (`text.secondary` + line-through when disabled), the
    `disabledBadge` `StatusChip`, a min-1 `NumberStepper` → `updatePerson`, an
    outlined `disable`/`enable` toggle → `updatePerson`, and a quiet danger
    `remove` → `useConfirmation()` → `removePerson`. All three writes are
    fire-and-forget (no status surface), per §5 quirk 5.
  - `AddPersonForm` — local `name`/`note`/`pagesPerDay`/`error` state; `onAdd`
    reproduces the legacy `onAddPerson` (blank → `nameRequired`; duplicate →
    `nameTaken` via `isNameUnique`; else `addPerson` with a trimmed name, optional
    trimmed note, and `Math.max(1, …)` pages), then resets name/note/error while
    keeping the pages value.
  - Empty roster → `emptyRoster`; query with no match → `noMatches`.
- `AdminApp.tsx` — `AdminRouteContent` now routes `roster` → `AdminRosterPage`
  (the placeholder handles only khatmas/khatma/settings until RM-520–540).
- Tests: `tests/app/admin-roster.test.tsx` (8) — list/badge, search-filter +
  focus (P4), empty, no-match, stepper (+ floor), enable/disable, remove
  confirm/cancel, and add validation/success/reset.
