## Scope

Fourth Phase 5 task, taken alongside RM-520 in the same session. Migrate the
admin **Khatma detail** route (`#/khatmas/{id}`) into the RM-500 admin shell:
header, edit card, per-member management (active), controls, completed controls,
and series history.

Parity oracle:
[`REACT_MIGRATION_UI_INVENTORY.md`](../../../REACT_MIGRATION_UI_INVENTORY.md)
§3.4 (Khatma detail), §5 quirk 5 (fire-and-forget granularity), §5 quirk 6
(detail selects apply only on the action button), §5 quirk 3
(`startNext` has no confirmation).
Legacy source of truth:
[`src/ui/admin/pages/khatma.ts`](../../../src/ui/admin/pages/khatma.ts).

In scope:

- **Header:** donut + title + status badge + facts line (remaining · round ·
  last distribution) + progress bar (completed = 100).
- **Edit card:** series name, number, created date; `saveKhatma` →
  `renameSeries` (whole series) + `updateKhatma` (number/date); status `saved`.
- **Members card (active):** per member — warning chip + `clearWarning` (clears
  the streak across the series' active khatmas) when flagged; chunk chip (state
  text + span) toggling `markRoundDone`/`clearRoundDone`; `returnToPool` (pending
  only) → confirm → `releaseMemberChunk`; `removeFromKhatma` → confirm →
  `removeMemberFromKhatma`; capacity editor (pages + surah-name select + juz) →
  `saveCapacity` → `updateKhatma`; add-member row → `addMemberToKhatma`. Loading:
  members list `common.loading` until assignments arrive.
- **Controls card (active):** reciter select → `updateKhatma(duaReciterId)`;
  `startNext` → prefilled next-in-series create form (jumps to `#/khatmas`, no
  confirmation, quirk 3); `markComplete` → confirm → `completeKhatma`; `remove`
  → confirm → `deleteKhatma`.
- **Completed controls:** only `startNext`.
- **History card:** series' completed khatmas; active-with-none → `noCompleted`;
  line: title · `completedOn` date · reciter.
- **Not found:** khatmas empty → `common.loading`; loaded but missing →
  `noActive`.

Out of scope: Khatmas list/create (RM-520, sibling task — but the shared
`CreateKhatmaPrefill` context and `useSurahs` hook land in RM-520 and are reused
here), Settings (RM-540), the draft-stability proof (RM-550), the full admin
integration suite (RM-560). RM-530 ships focused detail tests.

## Approach

- New `src/app/admin/pages/KhatmaPage.tsx` (`AdminKhatmaPage`); wire it into
  `AdminApp`'s `AdminRouteContent` (`khatma` → it).
- Reuse shared primitives (`SurfaceCard`, `AppTextField`, `AppSelectField`,
  `AppButton`, `StatusChip`, `ProgressBar`, `DonutChart`, `NumberStepper`-free
  numeric fields), `useConfirmation()`, `useWriteOperation()` for every mutation,
  and `useSurahs()` (RM-520) for the capacity surah-name selects.
- **Quirk 6 (apply-on-button):** the capacity editor and add-member fields are
  controlled local state whose values are written only when the action button is
  pressed — reproducing the legacy uncontrolled "read `.value` at click"
  semantics. The reciter select writes on change (fire-and-forget, quirk 5),
  matching the actual legacy `controlsCard` code.
- **Edit-card draft** is component-local state seeded once from the khatma, so a
  snapshot that updates `seriesName` never clobbers an in-progress edit (P2).
- `startNext` requests the shared `CreateKhatmaPrefill` (RM-520) then navigates
  to `#/khatmas`.

## Acceptance criteria

- Header/edit/members/controls/history render per §3.4; completed khatmas show
  only history + completed controls.
- Edit save renames the series and updates number/date, then shows `saved`.
- Member controls drive the correct writes; `returnToPool` only shows for a
  pending chunk; destructive actions confirm first.
- Capacity/add-member values apply only on their button (quirk 6); reciter writes
  on change.
- `startNext` opens the prefilled create form on `#/khatmas` with no confirmation.
- Not-found/loading states match the legacy copy.
- Typecheck, lint, and the full test suite pass (detail tests added).

## Intentional deltas (recorded per inventory §1.7 / §5)

1. **Confirmations** — every destructive/confirming action uses the shared
   RTL-safe confirmation dialog instead of `window.confirm`; copy and triggers
   are unchanged.
2. **Edit-save a11y/status** — the `saved` status carries `role="status"`; a
   save failure surfaces an error-tone `alert` (the legacy edit-save has no error
   surface). Behavior otherwise matches.
3. **Draft scope** — the edit-card draft is component-local/route-scoped (the
   RM-510 delta); it survives unrelated snapshots. RM-550 formalizes it.

## Implementation

- New `src/app/admin/pages/KhatmaPage.tsx` (`AdminKhatmaPage`), all hooks called
  before the not-found return so Rules of Hooks hold:
  - `HeaderCard` (donut + title + status + facts + `ProgressBar`), `EditCard`
    (name/number/date seeded once → `renameSeries` + `updateKhatma`, `saved` /
    `saveError` status), `MembersCard` (`common.loading` until assignments arrive,
    then a `MemberRow` per member + `AddMemberRow`).
  - `MemberRow`: warning `StatusChip` + `clearWarning` (across the series' active
    khatmas), a clickable chunk `ChunkChip` toggling `markRoundDone` /
    `clearRoundDone`, `returnToPool` (pending only, confirmed) →
    `releaseMemberChunk`, `removeFromKhatma` (confirmed) →
    `removeMemberFromKhatma`, and a `CapacityEditor`.
  - `CapacityEditor` / `AddMemberRow` hold controlled local state applied only on
    `saveCapacity` / `addMember` (**quirk 6**); `ControlsCard` writes the reciter
    on change (**quirk 5**), and `StartNextButton` requests the shared prefill and
    navigates to `#/khatmas` with no confirmation (**quirk 3**).
  - `HistoryCard` lists `completedInSeries` (active-with-none → `noCompleted`);
    not-found → `common.loading` (khatmas empty) or `noActive` (missing id).
- `AdminApp.tsx` routes `khatma` → `<AdminKhatmaPage id={route.id} />`; the P9
  open-detail subscription (`AdminAssignmentsSubscriptions`) already covered it.
- Reuses `useSurahs` / `SurahCapacitySelect` / `CreateKhatmaPrefill` from RM-520.
- Tests: `tests/app/admin-khatma.test.tsx` (11) — loading/not-found, header, edit
  save, chunk toggle, clear warning, return-to-pool (confirmed), capacity save,
  add member, complete + delete (confirmed), start-next prefill, and the
  completed-khatma layout. `admin-home.test.tsx`'s loader mock gained `getSurahs`
  (its P9 case now mounts the real detail page).
