## Scope

Sixth Phase 5 task, taken in the same session as RM-540 (it depends on RM-540's
settings draft existing). One consolidated proof that a live Firestore snapshot
arriving mid-interaction never clobbers an in-progress admin draft or closes an
open form — the parity-critical behaviors most likely to regress in a naive
React port (inventory §4, P2/P3/P4).

Parity oracle:
[`REACT_MIGRATION_UI_INVENTORY.md`](../../../REACT_MIGRATION_UI_INVENTORY.md)
§4 **P2** (admin drafts survive unrelated snapshots), **P3** (du3a touched
guard), **P4** (search caret/focus on re-render).

In scope — the four admin drafts and the search field:

- **P2 create form** (`#/khatmas`): the lifted create-form draft + member
  selection survive an unrelated khatmas snapshot; the form stays open.
- **P2 edit card** (`#/khatmas/{id}`): an in-progress series-name edit survives a
  concurrent rename snapshot (seeded once); the live header still reflects the
  incoming value.
- **P2 add-person** (`#/roster`): the add-person name draft survives a roster
  snapshot that adds another member.
- **P4 search** (`#/roster`): the search caret/focus and query survive a live
  roster snapshot, and the filter re-applies against the fresh data.
- **P3 du3a** (`#/settings`): proven in `admin-settings.test.tsx` (RM-540) — the
  same store-driven re-render these assert; referenced here to complete the
  matrix rather than duplicated.

Out of scope: new product behavior (this task adds no source changes — the
stability already exists from RM-500…RM-540; RM-550 pins it with tests), the full
admin integration suite (RM-560).

## Approach

- New `tests/app/admin-draft-stability.test.tsx`: mounts the real
  `AdminExperience` through the shared harness at each route, drives a partial
  interaction, then emits an **unrelated** live snapshot via the harness
  publishers (`subscriptions.khatmas/roster.emit`) and asserts the draft /
  focus / open-form state survives while the live surface still updates (so each
  case is a genuine re-render, not a no-op).
- No source changes: the drafts are already component-local/route-scoped
  (RM-510 delta) and the fields never remount, which is exactly what makes them
  snapshot-stable. RM-550 is the executable proof of that design.

## Acceptance criteria

- An unrelated snapshot does not overwrite or close: the create form, the edit
  card, or the add-person form (P2).
- Search caret/focus and query survive a live roster snapshot (P4).
- The du3a touched guard (P3) is covered (in `admin-settings.test.tsx`).
- Typecheck, lint, and the full test suite pass.

## Intentional deltas

None — this is a verification task. The behaviors asserted were introduced as the
RM-510 draft-scope delta and carried through RM-520/RM-530/RM-540; RM-550 records
no new deltas.
