## Reconciliation with the NOW.md steer

NOW.md preferred _extending_ the per-route suites over a new mega-file. That steer
guards against a catch-all that re-tests single-route states; it does not preclude
the integration layer. The remaining gaps are inherently cross-feature/temporal
(a khatma leaving the wanted set over time; a create write reflected on another
route; a distribution completing and reconciling listeners), so ‚Äî exactly as
RM-450 did for the member app ‚Äî they belong in a focused `admin-integration`
file, not bolted onto a single route's suite. No single-route regression gap
remained to fold back into a per-route suite (P7/P8/P9 were already present).

## Acceptance criteria

- **Critical CRUD across routes:** create a khatma on `#/khatmas`, have the write
  land via a live snapshot, follow its link into `#/khatmas/{id}`, and save a
  detail edit ‚Äî one continuous flow across khatmas-create ‚Üí khatmas-list ‚Üí
  khatma-detail (with a validation block-then-correct beat).
- **Distribution + realtime completion + listener release:** distribute on Home,
  then a completion snapshot drops the khatma from the active dashboard and
  releases its now-unwanted assignment listener (P10).
- **Listener reconciliation on navigation:** leaving a completed detail route for
  Home releases only that khatma's listener while the active khatma's listener is
  untouched (P9 ‚Üí P10, dynamic).
- **Error flow:** a failed distribution surfaces the generic error alert (not the
  same-day message, never a green success), re-enables the button, and a retry
  succeeds.

## Decisions and risks

- The suite mounts `<AdminExperience />` through `renderWithAppProviders` (the full
  admin tree minus the outer `AppProviders` the harness supplies) ‚Äî the
  established admin-suite pattern.
- The Quran loader (`@/content/quran/loader`) is mocked exactly as in the other
  admin suites so distribution scope maps and capacity surah names stay
  deterministic and offline; `range`-scope khatmas need no maps for their metrics.
- Navigation is driven by clicking the rendered detail/nav links (jsdom updates
  the hash and the router re-parses), mirroring `member-integration.test.tsx`.
- Roster write feedback is intentionally fire-and-forget (inventory ¬ß5 quirk 5),
  so the error-flow scenario uses distribution ‚Äî the only admin write that
  surfaces a status ‚Äî rather than a roster write.
- Scenarios cross feature boundaries the per-route suites do not, avoiding
  duplication of already-covered single-route states.
