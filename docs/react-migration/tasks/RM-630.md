# RM-630 — Optimize and document bundles

- Status: DONE
- Owner: Codex
- Dependencies: RM-620, RM-040 (complete)
- Baseline: completed RM-620 working tree at `cf11a52`

## Scope

Review member/admin entry and route imports after legacy cleanup, introduce
useful lazy-loading boundaries without changing route behavior, and record final
production initial-load sizes against the RM-040 budgets.

## Acceptance criteria

- Member/admin route imports and shared chunks are reviewed for practical split
  points; chosen boundaries preserve loading/error behavior.
- The production build has no avoidable oversized shared entry chunk warning.
- Final initial JS gzip and transfer sizes are recorded for both surfaces and
  remain within the 350/400 kB member and 375/425 kB admin limits.
- Typecheck, lint, focused routing/application tests, production build, and the
  bundle-budget checker pass.

## Starting point

- Member: 341.62 kB initial JS gzip / 387.73 kB initial transfer.
- Admin: 342.32 kB initial JS gzip / 388.43 kB initial transfer.
- Both entries statically pull a shared 1,087.18 kB raw / 336.15 kB gzip
  `bootstrap` chunk; Vite warns above 500 kB raw.

## Decisions and risks

- Route splitting must not introduce a blank screen; lazy boundaries need a
  localized loading state and the existing error boundary remains active.
- Do not optimize by weakening the established bundle budgets or hiding a real
  warning with a higher threshold.
- The development-only preview aliases may continue to support stable manifest
  measurement, but final documentation must report deployable production sizes.

## Import and splitting review

- A route-lazy prototype kept each default route eager and split the remaining
  member/admin pages behind localized `LoadingState` fallbacks. It measured
  338.05 kB member / 338.07 kB admin initial JS gzip, only 3.57/4.25 kB below
  the baseline because shared React/MUI/Firebase code dominates the graph.
- The prototype also intentionally made every non-default route asynchronous;
  37 focused component assertions observed the new loading phase. Given the
  roughly 1% transfer gain and small route modules, that behavior/complexity
  tradeoff was rejected and the synchronous route contract was restored.
- The production graph instead gives MUI and Firebase stable vendor cache groups.
  Together with the shared React/application runtime, every emitted JS chunk is
  now below Vite's 500 kB raw warning threshold; no threshold was raised.
- The obsolete preview-only build config/script was removed. The budget command
  now builds the deployable inputs, reads `dist/.vite/manifest.json`, and reports
  the actual production entry graphs.
