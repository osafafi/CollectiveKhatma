## Rollback plan basis (why it is credible)

- **No data migration.** `src/domain`, `src/data`, `src/content/quran`,
  `firestore.rules`, and `firestore.indexes.json` are byte-unchanged between base
  `6992007` and HEAD `7ba078c` (re-verified this pass — none appear in the diff).
  Both the legacy and React UIs interoperate against the identical live schema, so
  a UI rollback needs no data or rule reversal.
- **Concrete restore point.** `6992007` is the previously-deployed, known-good
  legacy app — a real commit, not a reconstruction.
- **Static deployment.** [`deploy.yml`](../../.github/workflows/deploy.yml) builds
  the static site and deploys identical Firestore rules on push to `main`; there is
  no server/DB state. Rollback = one revert commit + a Pages redeploy (minutes).
- **Merge-method input to OD-04.** A merge commit or squash yields a single
  revertable commit; pure fast-forward does not. The plan recommends the former for
  rollback ease.
